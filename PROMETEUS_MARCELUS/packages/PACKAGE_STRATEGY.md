# Package strategy - Prometeus Marcelus

Data: 2026-07-02

## Decisione

Per ora e meglio creare un **package editoriale/operativo** del progetto, non un GitHub Package tecnico.

GitHub Packages e pensato soprattutto per distribuire artefatti software come:

- container;
- pacchetti npm;
- pacchetti Maven;
- pacchetti NuGet;
- pacchetti Ruby;
- immagini o librerie installabili.

Prometeus Marcelus oggi e soprattutto:

- metodo;
- prompt;
- documentazione;
- casi reali;
- checklist;
- Custom GPT;
- biblioteca operativa.

Quindi il formato piu corretto, in questa fase, e uno **Starter Kit** pubblicabile come cartella e poi come Release GitHub.

## Package consigliato adesso

Nome:

```text
Prometeus Marcelus Starter Kit v0.1
```

Scopo:

Offrire a una persona nuova un percorso breve per capire, provare e testare Prometeus Marcelus senza leggere tutta la repository.

Contenuto:

- Start Here;
- Prompt master;
- metodo operativo;
- checklist prudenza;
- casi reali prioritari;
- Promethea Economica v0.1;
- routine GitHub;
- piano X free;
- istruzioni Custom GPT V4.

## Cosa NON includere

Non includere:

- materiali premium;
- monetizzazione riservata;
- dati personali;
- casi reali identificabili;
- file privati;
- prompt non verificati;
- documenti legali/medici/finanziari presentati come consulenza.

## Quando usare GitHub Packages vero

Usarlo solo quando nasceranno strumenti tecnici installabili, per esempio:

- una CLI `prometeus`;
- un pacchetto npm per validare prompt;
- un container per generare indici;
- un tool locale per controllare link, prompt e checklist;
- una piccola app web collegata al metodo.

Prima di allora, la sezione Packages puo restare vuota senza essere un problema.
